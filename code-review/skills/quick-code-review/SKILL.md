# quick-code-review

Быстрый однопроходный ревью. Проверяет все ключевые аспекты **в одном свипе**, без файлов и суб-агентов.
Результат — прямо в чат, компактной таблицей.

## Вход

1. **Явный путь** в запросе → читай файл/папку инструментом `Read`.
2. **Фрагмент кода** в запросе → ревьюй его.
3. **По умолчанию** — git diff текущей ветки:
   ```bash
   BASE=$(git merge-base HEAD main 2>/dev/null || git merge-base HEAD master 2>/dev/null)
   [ -n "$BASE" ] && git diff "$BASE"...HEAD || git diff HEAD
   ```

## Что проверять (все аспекты за один проход)

Смотри сразу по всем линзам — не переключайся между ними последовательно, замечай проблемы попутно:

| Линза | На что смотреть |
|---|---|
| **Баги / логика** | null/undefined без guard, off-by-one, перепутанные операторы, потерянный await, мутация входных аргументов |
| **Безопасность** | инъекции (SQL/команды), XSS, хардкод секретов, IDOR, отсутствие авторизации |
| **Типы** | `any`, небезопасные касты, несоответствие сигнатур, пропущенные null-проверки в TypeScript |
| **Производительность** | N+1 запросы, лишние ре-рендеры, блокирующие операции в цикле, отсутствие мемоизации там, где очевидно нужна |
| **Читаемость / KISS** | имена, скрытая логика без пояснений, функция делает больше одного; излишние абстракции и уровни indirection (оверинжиниринг «на будущее») |
| **DRY** | дублирование кода — одна и та же логика в двух+ местах, которую стоит вынести |
| **Контракты** | API-контракт нарушен, отсутствует валидация входа на границе системы |

Ищи только реальные проблемы. Не придумывай находки ради полноты таблицы.

## Северити

- 🔴 **Blocker** — баг в проде, уязвимость, падение
- 🟠 **Major** — неправильное поведение, серьёзный запах
- 🟡 **Minor** — субоптимально, но не критично
- ⚪ **Nit** — стиль, переименование

Сообщай только 🔴 и 🟠 по умолчанию. Если пользователь просит «всё» или «включая nit» — добавляй 🟡 и ⚪.

## Формат вывода

```
## Quick Review — [✅ PASS | ⚠️ WARN | ❌ FAIL]

| # | Файл | Метод / место | Проблема | Исправление |
|---|---|---|---|---|
| 🔴 | `auth/login.ts` | `validateToken()` | JWT не верифицируется — принимается любой payload | Добавить `jwt.verify(token, secret)` перед использованием |
| 🟠 | `api/users.ts` | `getUsers()` | N+1: запрос в цикле по каждому user | Один JOIN или `findMany({ where: { id: { in: ids } } })` |

_Найдено: 1 🔴 · 1 🟠 · 0 🟡_
```

Правила таблицы:
- Строка = одна конкретная находка. Не группируй разные проблемы в одну строку.
- **Файл** — только имя файла и номер строки, если известен (`login.ts:42`). Без полного пути.
- **Метод / место** — имя функции, компонента, хука, SQL-запроса. Если нет — `module level` или `import`.
- **Проблема** — одно предложение: что именно не так и почему это проблема.
- **Исправление** — конкретное действие или мини-сниппет (≤ 1 строки кода). Не «рефакторить», а «заменить X на Y».
- Строки сортируй: сначала 🔴, потом 🟠, 🟡, ⚪.

Если находок нет — пиши `✅ PASS` и одну строку «Явных проблем не найдено».

## Язык

Язык вывода — **язык запроса пользователя**. Если запрос без естественного языка (только код/diff) — **русский**.
Инструкции пользователя в запросе перекрывают эти дефолты.
