# explain-code

Анализирует конкретный участок кода и выдаёт структурированный отчёт: от назначения до неочевидных деталей.

## Когда использовать

- «Как работает `useAuth`?», «объясни этот компонент», «что делает этот хук»
- «Как запрос проходит от роутера до базы?», «покажи data flow»
- «Расскажи про этот модуль / сервис / middleware»
- «Walk me through X», «explain how X works», «trace this call»
- Нужно быстро разобраться в чужом или давно не тронутом коде

**Не использовать для:** поиска багов (→ `systematic-debugging`), добавления docstrings (→ `code-documenter`),
ревью качества (→ `code-reviewer`).

## Процесс

### 1. Определи цель

Из запроса извлеки цель — **что именно** надо объяснить:
- Конкретный файл / путь → читай его
- Функция / класс / хук по имени → сначала `grep -rn "name" src/` чтобы найти, потом читай
- Фича или концепция («как работает авторизация») → найди точку входа через grep/find, разверни граф

Если цель неоднозначна — уточни **одним вопросом** перед тем как читать код.

### 2. Разведка

Собери контекст не читая каждый файл целиком. Если есть CLAUDE.md в корне — прочитай его первым:
он даст карту слоёв, стека и конвенций до первого grep.

Остальные команды запускай параллельно:

```bash
# Найти все упоминания символа
grep -rn "TargetName" src/ --include="*.ts" -l

# Понять импорты / экспорты
grep -n "export\|import" path/to/file.ts | head -30

# Найти точки входа (роуты, контроллеры, обработчики)
grep -rn "router\.\|@Get\|@Post\|app\.\(get\|post\|use\)" src/ --include="*.ts" -l
```

Читай файлы **селективно** — только те части, которые нужны для ответа. Не читай весь репо.

### 3. Построй граф вызовов (при необходимости)

Если вопрос про поток данных или цепочку вызовов — пройди граф вручную:

```
вход (route / event / export)
  → middleware / guard / interceptor
  → handler / controller
  → service / use-case
  → repository / ORM / raw query
  → ответ / side-effect
```

Останавливайся на уровне детализации, которого достаточно для ответа.

### 4. Сформируй отчёт

Используй структуру ниже. Пропускай секции, которые не нужны или очевидны из контекста вопроса (не раздувай отчёт).

## Шаблон отчёта

```markdown
## [Название / путь к цели]

**Назначение**
Одно-два предложения: что это и зачем.

**Как работает**
Пронумерованный список шагов — от точки входа до результата.
Каждый шаг: конкретное действие + файл:строка, если неочевидно откуда.

1. `AuthGuard.canActivate()` (auth/auth.guard.ts:14) — проверяет наличие токена в заголовке
2. Если нет → 401, иначе передаёт управление в `JwtStrategy.validate()` (auth/jwt.strategy.ts:22)
3. ...

**Поток данных** *(только если про data flow)*
Вход → обработка → выход. Коротко, схема или список.

**Ключевые зависимости**
Что это использует: внешние модули, сервисы, конфиги.
- `prisma` (src/db/prisma.service.ts) — доступ к БД
- `ConfigService` — читает `JWT_SECRET` из env

**Неочевидное**
Всё, что удивит читателя: side-effects, порядок инициализации, edge cases, причины нестандартных решений.
Если всё очевидно — секцию не пиши.
```

## Правила

### ОБЯЗАТЕЛЬНО
- Привязывать утверждения к конкретным файлам и строкам (`file.ts:42`)
- Писать про **что происходит**, а не просто пересказывать имена (`вызывает validate()` → `validate() проверяет подпись JWT и возвращает payload`)
- Признавать неизвестное: «не нашёл вызывающую сторону» лучше, чем выдумывать

### НЕЛЬЗЯ
- Читать весь репо — только то, что нужно для вопроса
- Копировать код в отчёт без нужды — ссылка на строку лучше, чем вставка 50 строк
- Угадывать поведение без чтения кода
- Писать секции «Назначение», «Зависимости» если они очевидны из самого вопроса — шаблон адаптируется под нужды ответа

## Примеры

**Запрос:** «Как работает `useAuth` хук?»
→ Найти файл, прочитать хук, описать: что принимает, какой state держит, какие эффекты, что возвращает.

**Запрос:** «Объясни как запрос на `POST /orders` проходит через систему»
→ Найти роут, пройти цепочку middleware → controller → service → repo, описать поток.

**Запрос:** «Что делает `withRetry` в `src/lib/http.ts`?»
→ Прочитать функцию, описать алгоритм retry (условие, задержки, лимиты), отметить неочевидные детали.

## Язык

Отвечай на **языке запроса пользователя**. Если запрос без естественного языка (только путь к файлу или символ) — **русский**.
Инструкции пользователя в запросе перекрывают этот дефолт.
