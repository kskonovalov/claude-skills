# swagger-json

Строит OpenAPI 3.0 в виде отдельного **`.json` файла** для одного модуля Node.js.
На входе — **путь к модулю**. Цель: не выдумывать, а вытащить контракт API строго из кода.

> Нужен YAML вместо JSON → используй `swagger-yml`.
> Нужны inline `@swagger`-аннотации прямо в коде → используй `swagger-jsdoc`.

## Когда использовать

- «Сгенерируй swagger.json для `src/modules/users`»
- «Собери openapi-спеку в json для этого роутера»
- Нужен один самодостаточный JSON-файл с контрактом модуля (напр. для импорта в инструмент, ожидающий JSON)

**Не использовать для:** дизайна нового API с нуля (→ `api-designer`), ревью согласованности
контракта DTO↔Swagger (→ агент `code-review-contracts`).

## Вход

Один аргумент — **путь к модулю** (папка или файл роутера). Если путь не задан — спроси его
**одним вопросом** и останови работу до ответа. Если по пути нет роутов — сообщи и не выдумывай спеку.

## Процесс

Выполняй шаги строго по порядку, не пропускай.

### 1. Найти входную точку
- Найди файл роутера модуля: `*.router.js`, `*.routes.js`, `*.controller.*`, либо место с `app.use(...)` / `router.use(...)`.
- Зафиксируй базовый префикс пути (напр. `app.use('/api/users', ...)`).

```bash
grep -rn "router\.\(get\|post\|put\|patch\|delete\)\|app\.\(get\|post\|put\|patch\|delete\|use\)\|@\(Get\|Post\|Put\|Patch\|Delete\|Controller\)" <путь> | head -50
```

### 2. Собрать список роутов
Для каждого роута зафиксируй:
- HTTP-метод (`get` / `post` / `put` / `patch` / `delete`)
- путь (с учётом префикса из шага 1)

### 3. Разобрать параметры каждого роута
- **path** — параметры вида `/:id`
- **query** — обращения к `req.query.*`
- **body** — поля `req.body.*` + типы
- **headers** — токены авторизации и прочие заголовки

### 4. Определить ответы
- Коды статусов из `res.status(...)` / `res.sendStatus(...)`
- Структуру тела из `res.json(...)` / `res.send(...)`

### 5. Вытащить модели / схемы
Источники → в `components.schemas`:
- ORM-модели (Mongoose / Sequelize / Prisma)
- валидаторы (Joi / Zod / express-validator / class-validator DTO)
- TS-типы / интерфейсы

### 6. Определить безопасность
- Найди middleware авторизации (`authMiddleware`, `passport`, `@UseGuards`, проверка `Authorization`).
- Опиши в `components.securitySchemes` (напр. bearer JWT) и привяжи к защищённым роутам через `security`.

### 7. Собрать структуру OpenAPI
- `"openapi": "3.0.x"`, `info`, `servers`
- `paths` — из шагов 2–4
- `components.schemas` — из шага 5
- `components.securitySchemes` — из шага 6

### 8. Записать и проверить
- **Сохраняй артефакт внутри проекта, в папку AI-инструмента.** Определи её в корне проекта: `.claude/`, `.cursor/`,
  `.gigacode/` и т.п. — ту, что реально присутствует. Если их несколько — предпочти `.claude/`; если ни одной — спроси,
  куда сохранять, **одним вопросом**.
- Имя и путь файла: `<ai-tool>/swagger/<module-name>.swagger.json`, где `<module-name>` — имя модуля
  (напр. `.claude/swagger/users.swagger.json`). Не сохраняй вне проекта и не раскидывай по папкам модулей.
- Провалидируй любым актуальным OpenAPI-валидатором: `npx @redocly/cli lint <file>.json` (или подскажи открыть в Swagger Editor). Устаревший `swagger-cli` не использовать.

## Правила вывода

### ОБЯЗАТЕЛЬНО
- Выводить только **валидный JSON** (без trailing-запятых, без комментариев — JSON их не поддерживает).
- Если параметр/тип не найден явно — пометить значением-плейсхолдером в `description`: `"TODO: уточнить"`, **не выдумывать** реальный тип.
- Описывать только роуты, **которые есть в коде**. Ничего не добавлять «по аналогии».
- Привязывать `security` к роутам, где реально есть auth-middleware.

### НЕЛЬЗЯ
- Добавлять роуты, методы, поля или коды ответов, которых нет в коде.
- Угадывать типы body/query без подтверждения в коде или валидаторе.
- Вставлять `//`-комментарии в JSON (невалидно).
- Сканировать что-то за пределами указанного модуля (кроме явных импортов моделей/типов).

## Язык
Сообщения — на языке запроса пользователя, дефолт **русский**. Ключи JSON — стандартные OpenAPI (английские).
`description` полей можно на языке запроса.
