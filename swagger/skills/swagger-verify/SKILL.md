# swagger-verify

Верифицирует **уже существующее** OpenAPI/Swagger описание модуля. Не генерирует и не правит спеку —
только проверяет и выдаёт отчёт о расхождениях. Парная проверка к `swagger-yml` / `swagger-json` / `swagger-jsdoc`.

> Нужно **сгенерировать** спеку → `swagger-yml` / `swagger-json` / `swagger-jsdoc`.

## Когда использовать

- «Проверь, правильный ли `users.swagger.yml`»
- «Сверь swagger с кодом модуля `src/modules/document`»
- «Валиден ли этот openapi.json и не выдуман ли там роут»
- После генерации спеки одним из swagger-скиллов — как контроль качества

**Не использовать для:** дизайна нового API (→ `api-designer`), ревью согласованности контракта во всём
проекте (→ агент `code-review-contracts`).

## Вход

Два ориентира:
1. **Что проверять** — путь к артефакту (`.yml` / `.json`) **или** к модулю с inline `@swagger`-блоками.
2. **С чем сверять** — путь к исходному модулю (роутер + модели).

Если задан только артефакт — найди соответствующий модуль (по имени/`servers`/`tags`); если не нашёл — спроси
**одним вопросом**. Для inline-варианта артефакт и модуль — это один и тот же код. Если нечего проверять
(нет файла/блоков) — сообщи и остановись.

## Определи формат

- Файл `.yml` / `.yaml` → проверяй как YAML-спеку.
- Файл `.json` → проверяй как JSON-спеку.
- Модуль с блоками `/** @swagger ... */` → собери из них спеку (как это делает `swagger-jsdoc` при сборке)
  и проверяй её. Если в проекте есть конфиг `swagger-jsdoc` — используй его сборку.

## Процесс

Выполняй шаги по порядку.

### 1. Формальная валидность (синтаксис + схема OpenAPI)
- **JSON**: убедись, что это валидный JSON (нет trailing-запятых, нет `//`-комментариев).
- **YAML**: убедись, что YAML парсится (отступы, отсутствие табов).
- Прогони OpenAPI-валидатор: `npx @redocly/cli lint <file>` (для inline — на собранной спеке).
  Устаревший `swagger-cli` не использовать.
- Зафиксируй: версия `openapi: 3.0.x`, наличие `info`, `paths`, корректность `$ref` (все ссылки разрешаются).

### 2. Сверка роутов: спека ↔ код
Собери список роутов из кода (метод + путь с префиксом) и из спеки, сравни:

```bash
grep -rn "router\.\(get\|post\|put\|patch\|delete\)\|app\.\(get\|post\|put\|patch\|delete\)\|@\(Get\|Post\|Put\|Patch\|Delete\)" <модуль>
```

- **Выдуманные** — роут есть в спеке, но нет в коде → ошибка.
- **Потерянные** — роут есть в коде, но нет в спеке → ошибка.
- **Несовпадение пути/метода** (`:id` vs `{id}`, неверный префикс) → ошибка.

### 3. Сверка деталей каждого совпавшего роута
- **path/query/body/headers**: параметры спеки совпадают с `req.params`/`req.query`/`req.body`/заголовками в коде.
- **ответы**: коды из спеки совпадают с `res.status(...)`; описанные тела соответствуют `res.json(...)`.
- **security**: `security` стоит ровно там, где в коде есть auth-middleware (и не стоит, где его нет).

### 4. Сверка моделей / схем
- `components.schemas` совпадают с реальными ORM-моделями / валидаторами / TS-типами: поля и типы те же,
  нет лишних/недостающих полей.
- Все `$ref` указывают на существующие схемы.

### 5. Остатки-заглушки
- Найди незакрытые `TODO: уточнить` (в YAML — `# TODO`, в JSON — в `description`). Каждый — повод дозаполнить.

## Отчёт

Выводи результат **в чат** компактной таблицей, без правки файлов:

```markdown
## Верификация: <артефакт> ↔ <модуль>

**Формальная валидность:** OK / FAIL (<кратко: что не так>)

| # | Серьёзность | Где (роут / схема) | Проблема | Ожидалось по коду |
|---|-------------|--------------------|----------|-------------------|
| 1 | 🔴 ошибка   | POST /users        | роут в спеке, в коде нет        | удалить из спеки |
| 2 | 🔴 ошибка   | GET /users/:id     | путь `:id`, должно быть `{id}`  | `{id}` |
| 3 | 🟡 расхожд. | User.email         | в спеке `number`, в коде string | `type: string` |
| 4 | ⚪ заглушка  | GET /orders        | `TODO: уточнить` в `description` | дозаполнить тип ответа |

**Итог:** N ошибок, M расхождений, K заглушек. Спека <корректна / требует правок>.
```

Серьёзность: 🔴 ошибка (выдуманное/потерянное/неверный путь/метод), 🟡 расхождение (тип/код/параметр не совпал),
⚪ заглушка (`TODO`). Если всё чисто — так и напиши, таблицу можно опустить.

## Правила

### ОБЯЗАТЕЛЬНО
- Сверять **со фактическим кодом**, а не «на глаз»: каждое замечание привязывать к роуту/полю и месту в коде.
- Различать «нет в коде» (выдумано) и «нет в спеке» (потеряно) — это разные ошибки.
- Помечать незакрытые `TODO: уточнить` как незавершённость.

### НЕЛЬЗЯ
- Править артефакт или код — скилл только проверяет и сообщает (исправление → перегенерировать
  соответствующим swagger-скиллом).
- Считать ошибкой стилевые warning'и валидатора (missing `operationId`/`tags`), если структура валидна —
  помечать их максимум как ⚪, не как 🔴.
- Сканировать за пределами указанного модуля (кроме явных импортов моделей/типов).

## Язык
Отчёт — на языке запроса пользователя, дефолт **русский**. Ключи OpenAPI в примерах — английские.
