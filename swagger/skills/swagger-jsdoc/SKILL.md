# swagger-jsdoc

Не создаёт отдельный файл, а **встраивает** OpenAPI-описание в код в виде JSDoc-блоков
`/** @swagger ... */` над обработчиками роутов (формат библиотеки `swagger-jsdoc`).
На входе — **путь к модулю**. Описание берётся строго из кода.

> Нужен отдельный файл-спека → используй `swagger-yml` или `swagger-json`.

## Когда использовать

- «Добавь swagger-комментарии в `src/modules/document`»
- «Аннотируй роуты этого модуля для swagger-jsdoc»
- Проект уже собирает спеку через `swagger-jsdoc` (сканирует JSDoc по `apis: [...]`) и нужно описать новый модуль

**Не использовать для:** генерации самостоятельного `.yml`/`.json` (→ `swagger-yml` / `swagger-json`),
дизайна нового API (→ `api-designer`).

## Вход

Один аргумент — **путь к модулю** (папка или файл роутера). Если путь не задан — спроси его
**одним вопросом** и останови работу до ответа. Если по пути нет роутов — сообщи и ничего не добавляй.

## Процесс

Выполняй шаги строго по порядку, не пропускай.

### 1. Найти роуты и их обработчики
- Найди файл роутера: `*.router.js`, `*.routes.js`, `*.controller.*`, либо `app.use(...)` / `router.use(...)`.
- Зафиксируй базовый префикс (напр. `/api/document`) — он входит в `@swagger` путь.
- Для каждого роута запомни **точную строку объявления** (`router.get('/:uuid', ...)`) — над ней пойдёт блок.

```bash
grep -rn "router\.\(get\|post\|put\|patch\|delete\)\|app\.\(get\|post\|put\|patch\|delete\)" <путь>
```

### 2. Для каждого роута собрать данные (как для обычной спеки)
- **path**-параметры (`/:uuid` → `uuid`), **query** (`req.query.*`), **body** (`req.body.*` + типы), **headers**
- коды и тела ответов (`res.status(...)`, `res.json(...)`)
- auth-middleware → `security`

### 3. Сформировать JSDoc-блок над каждым обработчиком
Над строкой объявления роута вставь блок. Путь — полный, с префиксом и в OpenAPI-формате (`/document/{uuid}`, а не `/:uuid`):

```js
/**
 * @swagger
 * /document/{uuid}:
 *   get:
 *     summary: <кратко из имени/логики обработчика>
 *     tags: [Document]
 *     security:
 *       - bearerAuth: []          # только если есть auth-middleware
 *     parameters:
 *       - in: path
 *         name: uuid
 *         required: true
 *         schema:
 *           type: string
 *           format: uuid
 *     responses:
 *       200:
 *         description: OK
 *         content:
 *           application/json:
 *             schema:
 *               $ref: '#/components/schemas/Document'
 *       404:
 *         description: Not found
 */
router.get('/:uuid', authMiddleware, getDocument);
```

- Для `post`/`put`/`patch` добавляй `requestBody` с `content.application/json.schema`.
- Тип не найден явно → `# TODO: уточнить` в комментарии строки, **не выдумывай**.

### 4. Описать переиспользуемые схемы (один раз на модуль)
Модели/DTO/типы из кода вынеси в один блок `components.schemas` (обычно над роутером или в `*.swagger.js`).
Если блок вынесен в отдельный файл (`*.swagger.js`) — этот файл тоже должен попадать в `apis: [...]` конфига
`swagger-jsdoc`, иначе схемы не будут просканированы (см. шаг 5).

```js
/**
 * @swagger
 * components:
 *   schemas:
 *     Document:
 *       type: object
 *       properties:
 *         uuid: { type: string, format: uuid }
 *         # ... только реальные поля модели
 *   securitySchemes:
 *     bearerAuth:
 *       type: http
 *       scheme: bearer
 *       bearerFormat: JWT
 */
```

### 5. Вставить и проверить
- Внеси блоки в файл(ы) через редактирование — **над** соответствующими объявлениями, не ломая код.
- Убедись, что путь модуля попадает в `apis: [...]` конфига `swagger-jsdoc`; если нет — подскажи добавить.
- При наличии сборки спеки — прогони её и проверь, что блоки парсятся без ошибок.

## Правила вывода

### ОБЯЗАТЕЛЬНО
- Аннотировать только роуты, **которые есть в коде**; блок — ровно над их объявлением.
- Путь в `@swagger` — полный (с префиксом) и в OpenAPI-формате: `{uuid}`, не `:uuid`.
- Незнакомый тип/код → `# TODO: уточнить`, не подставлять выдуманное.
- Переиспользуемые модели — через `$ref`, описывать схему один раз.

### НЕЛЬЗЯ
- Менять логику кода — только добавлять комментарии-блоки.
- Добавлять роуты, поля, коды ответов, которых нет в коде.
- Дублировать `components.schemas` у каждого роута — выносить один раз.

## Язык
Сообщения — на языке запроса, дефолт **русский**. Ключи OpenAPI — английские; `summary`/`description`
можно на языке запроса.
