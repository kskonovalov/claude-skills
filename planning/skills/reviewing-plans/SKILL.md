# reviewing-plans

Критический ревьюер **написанного плана**. Цель — найти слабые места до того, как начнётся реализация.
Запускается после `writing-plans`, до `executing-plans`.

**Announce at start:** "I'm using the reviewing-plans skill to critique the plan."

## Вход

1. **Явный путь** к плану в запросе → читай этот файл.
2. **По умолчанию** — последний файл из `docs/plans/` (по дате в имени).
3. **Спек** — если есть `docs/specs/`, найди спек, который соответствует плану (по теме/дате), и читай его тоже.

До анализа прочитай CLAUDE.md проекта (стек, архитектура, конвенции).

## Что проверять

### 1. Покрытие спека
Сопоставь каждое требование/раздел спека с задачами плана. Есть ли требование без задачи?
Есть ли задача без требования в спеке (scope creep)?

### 2. Плейсхолдеры и расплывчатость
Ищи в шагах:
- «TBD», «TODO», «implement later», «fill in details»
- «Add appropriate error handling» / «handle edge cases» без кода
- «Similar to Task N» вместо реального кода
- Шаги, описывающие *что* делать без *как* (нет кода там, где он нужен)
- Команды без ожидаемого вывода

### 3. Технические риски
- Предположения о внешних зависимостях/API без проверки (например, «предполагаем, что endpoint X существует»)
- Шаги, которые могут упасть из-за неизвестного поведения среды
- Миграции/изменения схемы БД без rollback-стратегии
- Интеграции с третьими сторонами без mock/stub-стратегии на время разработки

### 4. Тесты
- Каждая задача с кодом имеет тест (TDD: тест до реализации)?
- Тесты не тривиальны (не просто «проверяем, что функция существует»)?
- Покрыты ли edge cases и error paths — не только happy path?
- Используется ли реальный тест-раннер проекта (не выдуманный)?

### 5. Размер и порядок задач
- Задачи действительно по 2–5 минут, или есть монстр-задачи на час?
- Нет ли скрытых зависимостей (задача 5 требует типа, определённого только в задаче 7)?
- Порядок задач логичен: фундамент раньше надстройки?

### 6. Консистентность типов и имён
- Функции, типы, пропсы, методы имеют одно и то же имя по всему плану?
- Типы, используемые в поздних задачах, определены в ранних?

### 7. Архитектура и YAGNI
- Нет ли оверинжиниринга «на будущее» (паттерны/абстракции без текущей необходимости)?
- Соответствует ли структура файлов архитектурным слоям проекта (CLAUDE.md)?
- Нет ли нарушений принятых в проекте конвенций?

### 8. Коммиты
- Каждая завершённая задача заканчивается коммитом?
- Коммиты атомарны (один связный набор изменений, не «всё подряд»)?

## Северити находок

- 🔴 **Blocker** — план нельзя исполнить без исправления (пропущено требование, сломан порядок зависимостей, плейсхолдер вместо реального шага)
- 🟠 **Major** — риск провала или значительного рефакторинга после выполнения (нет тестов, технический риск без митигации, монстр-задача)
- 🟡 **Minor** — снизит качество или скорость, но не заблокирует (непоследовательные имена, отсутствие edge-case теста)
- ⚪ **Suggestion** — можно сделать лучше, но необязательно

## Формат вывода

Выводи прямо в чат (файл не пишется — план ещё не финальный):

```
## Plan Review — [✅ READY | ⚠️ NEEDS FIXES | ❌ BLOCKED]

| # | Задача плана | Категория | Проблема | Рекомендация |
|---|---|---|---|---|
| 🔴 | Task 3 | Плейсхолдер | Шаг «Add error handling» без кода | Написать конкретный try/catch с типом ошибки |
| 🟠 | Task 5 | Тесты | Нет теста для error path в `createUser` | Добавить тест с мок-ошибкой БД |
| 🟡 | Task 2, 6 | Имена | `clearLayer()` в Task 2, `clearFullLayer()` в Task 6 | Привести к единому имени |

_Найдено: 1 🔴 · 1 🟠 · 1 🟡_

### Итог
<вердикт + что главное исправить перед стартом>
```

Правила таблицы:
- Строка = одна конкретная находка.
- **Задача плана** — номер/название задачи из плана (или «Global», если касается всего плана).
- **Категория** — одно из: Покрытие спека / Плейсхолдер / Тех. риск / Тесты / Размер/порядок / Имена / Архитектура / Коммиты.
- **Проблема** — одно предложение: что именно не так.
- **Рекомендация** — конкретное действие: что дописать, переименовать, добавить.

Если находок нет — `✅ READY` и одна строка «План готов к исполнению».

## После ревью

Если есть 🔴 Blocker-и:
- Предложи исправить план прямо сейчас (инлайн правка с помощью `Write`/`Edit`) — и затем передай в `executing-plans`.

Если только 🟠/🟡:
- Спроси пользователя: «Исправить эти замечания перед стартом или запускать как есть?»

Если ✅ READY:
- Сразу предложи передать в `executing-plans`.
