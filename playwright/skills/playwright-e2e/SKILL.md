# playwright-e2e

Строит **блок E2E-тестов Playwright** для одного React-компонента по **пути к файлу**. Цель — не выдумывать,
а вытащить проверяемое поведение **строго из кода**: тест-атрибуты, обработчики, состояния, условный рендер.
Результат — один `test.describe.only(...)` блок, который пользователь **вставляет в свой Playwright-спек** и
расширяет покрытие.

> Окружение считается **уже настроенным** (baseURL, `testIdAttribute`, импорты `test`/`expect`, навигация в
> `beforeEach` — всё на стороне пользователя). Скилл **не** запускает тесты, **не** ищет `playwright.config`,
> **не** поднимает сервер и **не** добавляет импорты/шапку файла.

## Когда использовать

- «Напиши playwright-тесты на `src/components/UserForm.tsx`»
- «Покрой этот компонент e2e, проверь разные комбинации пропсов»
- Есть компонент с тест-атрибутами, нужен стартовый набор сценариев для расширения

**Не использовать для:** настройки Playwright с нуля, component-тестов в изоляции
(`@playwright/experimental-ct-react`), unit-тестов логики (→ Jest/Vitest), ревью существующих тестов.

## Вход

- **Путь к компоненту/файлу** — обязательный. Если не задан — спроси **одним вопросом** и остановись.
- **Сценарии словами** — опционально (напр. «проверь сабмит с пустыми полями», «проверь разные комбинации»).
  Пользовательские сценарии имеют приоритет и **всегда** попадают в итог.

Если по пути нет React-компонента (или файл не найден) — сообщи и не выдумывай тесты.

## Допущения (не проверять, не искать)

- **На всех нужных элементах есть `data-test-id`.** Проп `testId` в React на странице становится
  `data-test-id`. Значит: адресуй всё через `getByTestId(<значение>)`, fallback на `getByRole`/`getByText`
  **не нужен**, `// TODO: добавить data-test-id` не пиши.
- Playwright уже настроен. Не читай/не упоминай `playwright.config`, `testIdAttribute`, импорты.
- Не ищи роуты по кодбазе и не пересказывай файл целиком — только извлекай тест-id и поведение.

## Процесс

Выполняй шаги строго по порядку, не пропускай.

### 1. Прочитать компонент
- Прочитай файл по пути. Зафиксируй имя компонента, пропсы и что он рендерит.
- Не пересказывай файл в чат — он нужен только как источник тест-id и поведения.

### 2. Собрать тест-атрибуты
Найди значения тест-id всех адресуемых элементов (в DOM это `data-test-id`; в коде может быть записано
как `data-test-id`, `data-testid` или проп `testId`/`testID`):

```bash
grep -nE "data-test-id|data-testid|testId|testID" <путь-к-файлу>
```

Для каждого зафиксируй: значение id, что за элемент/роль, какой обработчик привязан (`onClick`/`onSubmit`/`onChange`).

### 3. Вывести сценарии из кода
Собери кандидатные сценарии из того, что подтверждено кодом:
- **render/visibility** — ключевые элементы по собранным тест-id видимы
- **взаимодействия** — `onClick` → click, `onSubmit` → заполнение+submit, `onChange` → fill/select
- **состояния** — `disabled` / `loading` / `error` / пустое состояние, если есть в разметке
- **условный рендер** — ветки `&&`, `? :`, ранние `return null`
- **комбинации пропсов** — только если пользователь просил «разные комбинации»: 2-4 значимых набора
  (не комбинаторный взрыв)
- **пользовательские сценарии** — добавь как есть, они приоритетны

### 4. Показать список сценариев и дождаться подтверждения (HARD-GATE)
- Выведи **в чат** пронумерованный список кандидатных сценариев (кратко: что проверяем).
- Отметь, что выведено из кода, а что добавлено по просьбе пользователя.
- **Остановись** и дождись «ок» / правок. Не пиши тесты до подтверждения.

### 5. Сгенерировать блок
Выдай **ровно один** блок и больше ничего (без импортов, без шапки, без пояснений вне кода):

```ts
test.describe.only("<коротко: что проверяем>", () => {
  test("<сценарий 1>", async ({ page }) => {
    // ...
  });
  // остальные подтверждённые сценарии
});
```

Правила внутри блока:
- Адресация элементов — **только `getByTestId(<значение>)`**.
- Ассерты — web-first (`await expect(locator).toBeVisible()`, `.toBeDisabled()`, `.toHaveText(...)`), всегда `await`.
- Навигация считается сделанной окружением (`beforeEach`). Добавляй `page.goto(...)` **только** если сценарий
  явно этого требует, и тогда `// TODO: роут` вместо выдуманного пути.
- Неизвестные ожидаемые значения (тексты, URL, кол-во) — `// TODO: уточнить`, **не выдумывать**.

### 6. Сохранить
- **По умолчанию** — в папку AI-инструмента внутри проекта:
  `.gigacode/playwright/<component-name>/<component-name>.spec.ts` (`<component-name>` — kebab-case от имени
  компонента, напр. `user-form`). В файл кладётся **только** сгенерированный `describe.only`-блок.
- Создай недостающие папки. Если пользователь указал другой путь — сохрани туда. Не сохраняй вне проекта.
- Сообщи финальный путь.

## Правила вывода

### ОБЯЗАТЕЛЬНО
- Тестировать только элементы/обработчики/состояния, **которые есть в коде**.
- Адресовать всё через `getByTestId()`.
- Включать все сценарии, описанные пользователем.
- Помечать `// TODO` то, что не подтверждено кодом (роут, ожидаемые значения).
- Выдавать корректный TypeScript и Playwright test API.

### НЕЛЬЗЯ
- Придумывать тест-id, роуты, тексты или элементы, которых нет в коде.
- Добавлять импорты, шапку файла, комментарии про конфиг/`testIdAttribute`, обёртки вокруг `describe.only`.
- Искать/читать `playwright.config`, роуты по кодбазе; пересказывать файл целиком.
- Создавать README, документацию, `.md`-файлы или любые артефакты, кроме указанного `.spec.ts` с блоком.
- Писать тесты до подтверждения списка сценариев (шаг 4).
- Раздувать комбинации пропсов до комбинаторного взрыва.

## Язык
Сообщения и список сценариев — на языке запроса, дефолт **русский**. Код теста — на английском; названия
`test(...)`/`describe.only(...)` можно на языке запроса, если так просил пользователь.
