# playwright-e2e

Строит **E2E-тесты Playwright** для одного React-компонента по **пути к файлу**. Цель — не выдумывать,
а вытащить проверяемое поведение **строго из кода**: тест-атрибуты, обработчики, состояния, условный
рендер. Результат — готовые `*.spec.ts`, которые пользователь **копипастит в свой Playwright** и расширяет
покрытие.

> Это E2E против **уже настроенного окружения** (baseURL / dev-сервер заданы в проекте). Скилл **не**
> запускает тесты, **не** правит `playwright.config`, **не** поднимает сервер — только генерирует файлы.
> Нужны component-тесты в изоляции (`@playwright/experimental-ct-react`) — это другой сценарий, не сюда.

## Когда использовать

- «Напиши playwright-тесты на `src/components/UserForm.tsx`»
- «Покрой этот компонент e2e, проверь разные комбинации пропсов»
- Есть компонент с `data-test-id` / `testId`, нужен стартовый набор спек для расширения

**Не использовать для:** дизайна тест-стратегии всего проекта, настройки Playwright с нуля, unit-тестов
логики (→ Jest/Vitest), ревью существующих тестов.

## Вход

- **Путь к компоненту/файлу** — обязательный. Если не задан — спроси **одним вопросом** и остановись.
- **Сценарии словами** — опционально (напр. «проверь сабмит с пустыми полями», «проверь разные комбинации»).
  Пользовательские сценарии имеют приоритет и **всегда** попадают в итог.

Если по пути нет React-компонента (или файл не найден) — сообщи и не выдумывай тесты.

## Процесс

Выполняй шаги строго по порядку, не пропускай.

### 1. Прочитать компонент
- Прочитай файл по указанному пути целиком.
- Зафиксируй имя компонента (для имени папки/файла), его пропсы и то, что он рендерит.

### 2. Собрать тест-атрибуты
Найди все элементы, адресуемые по тест-id, **которые реально есть в коде**:

```bash
grep -nE "data-test-id|data-testid|testId|testID" <путь-к-файлу>
```

Для каждого зафиксируй:
- значение id (напр. `submit-button`)
- на каком элементе / какой роли (`button`, `input`, `form`, ...)
- какой обработчик привязан (`onClick`, `onSubmit`, `onChange`, ...) — если есть

> Playwright `getByTestId()` по умолчанию читает `data-testid`. Если в DOM попадает `data-test-id`
> (или иной атрибут) — в спеку добавь комментарий-напоминание про `testIdAttribute` (см. шаг 6).
> Если у нужного элемента **нет** тест-id — не выдумывай его: используй `getByRole`/`getByText`
> и пометь `// TODO: добавить data-test-id`.

### 3. Определить роут (best-effort)
- Поищи, где компонент используется и на каком роуте он отрисован:
  - `react-router`: `<Route path="..." element={<Component/>} />`
  - Next.js: путь роутинга из расположения файла (`app/`/`pages/`)
  - импорты/использования компонента в родителях
- Нашёл однозначно — используй `page.goto('<route>')`.
- **Не нашёл или неоднозначно** — не блокируйся: подставь `page.goto('/TODO')` и рядом
  `// TODO: укажи роут, где отрисован <Component>`. Не выдумывай реальный путь.

### 4. Вывести сценарии из кода
Собери кандидатные сценарии из того, что подтверждено кодом:
- **render/visibility** — ключевые элементы по собранным тест-id видимы после захода на страницу
- **взаимодействия** — `onClick` → click, `onSubmit` → заполнение+submit, `onChange` → fill/select
- **состояния** — `disabled` / `loading` / `error` / пустое состояние, если они есть в разметке
- **условный рендер** — ветки `&&`, `? :`, ранние `return null`
- **комбинации пропсов** — только если пользователь просил «разные комбинации»: перечисли
  осмысленные наборы пропсов, влияющие на разметку/поведение (не комбинаторный взрыв — 2-4 значимых)
- **пользовательские сценарии** — добавь как есть, они приоритетны

### 5. Показать список сценариев и дождаться подтверждения (HARD-GATE)
- Выведи **в чат** пронумерованный список кандидатных сценариев (кратко: что проверяем).
- Отметь, что выведено из кода, а что добавлено по просьбе пользователя.
- **Остановись** и дождись «ок» / правок. Не пиши файлы до подтверждения.
- Если пользователь правит список — обнови и подтверди ещё раз при существенных изменениях.

### 6. Сгенерировать спеку
- Один `test.describe('<Component>', ...)`, внутри — по `test(...)` на каждый подтверждённый сценарий.
- Адресация элементов — **через `getByTestId()`**.
- Ассерты — web-first (`await expect(locator).toBeVisible()`, `.toBeDisabled()`, `.toHaveText(...)` и т.п.),
  всегда `await`.
- Неизвестные ожидаемые значения (тексты, URL, кол-во) — `// TODO: уточнить`, **не выдумывать**.
- В начало файла — шапка-комментарий:
  ```ts
  // Playwright E2E для <Component>. Требует настроенного baseURL и, если тест-id — не data-testid,
  // укажи в playwright.config: use: { testIdAttribute: 'data-test-id' }
  ```

### 7. Сохранить
- **По умолчанию** — в папку AI-инструмента внутри проекта: `.gigacode/playwright/<component-name>/<component-name>.spec.ts`.
  `<component-name>` — kebab-case от имени компонента (напр. `user-form`).
- Создай недостающие папки. Если пользователь указал другой путь — сохрани туда.
- Не сохраняй вне проекта. Сообщи финальный путь и краткое содержание.

## Правила вывода

### ОБЯЗАТЕЛЬНО
- Тестировать только элементы/обработчики/состояния, **которые есть в коде**.
- Использовать `getByTestId()` для собранных тест-id.
- Включать все сценарии, описанные пользователем.
- Помечать `// TODO` всё, что не подтверждено кодом (роут, ожидаемые значения, отсутствующий тест-id).
- Выдавать валидный TypeScript и корректный Playwright test API (`@playwright/test`).

### НЕЛЬЗЯ
- Придумывать тест-id, роуты, тексты или элементы, которых нет в коде.
- Писать файлы до подтверждения списка сценариев (шаг 5).
- Запускать тесты, менять `playwright.config` или поднимать сервер.
- Раздувать комбинации пропсов до комбинаторного взрыва — только значимые наборы.

## Язык
Сообщения и список сценариев — на языке запроса, дефолт **русский**. Код теста — на английском;
названия `test(...)`/`describe(...)` можно на языке запроса, если так просил пользователь.
